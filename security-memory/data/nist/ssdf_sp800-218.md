# NIST Secure Software Development Framework (SSDF) SP 800-218

**Source:** https://csrc.nist.gov/publications/detail/sp/800-218/final
**Version:** 1.1 (February 2022)

The SSDF defines a set of fundamental, sound, and secure software development practices based on
established secure software development practice documents from BSA, OWASP, SAFECode, and others.

---

## PO — Prepare the Organization

Ensure the organization has the people, processes, and technology to develop secure software.

### PO.1 — Define Security Requirements for Software Development
Ensure that security requirements are defined and consistently applied throughout all SDLC practices.

**Tasks:**
- PO.1.1: Identify and document all security requirements for the organization's software development infrastructures and processes, and maintain the requirements over time.
- PO.1.2: Identify and document all security requirements for software projects, and maintain the requirements over time.
- PO.1.3: Communicate requirements to all third parties who will provide commercial software components to the organization for reuse by the organization's own software.

### PO.2 — Implement Roles and Responsibilities
Ensure that everyone participating in the SDLC is prepared to perform their security-related roles and responsibilities.

**Tasks:**
- PO.2.1: Create new roles and alter responsibilities for existing roles as needed to encompass all parts of the SDLC.
- PO.2.2: Provide role-based training for all personnel with responsibilities that contribute to secure development.
- PO.2.3: Obtain upper management commitment to secure development.

### PO.3 — Implement Supporting Toolchains
Use automation to reduce human effort and improve consistency and effectiveness of security practices.

**Tasks:**
- PO.3.1: Specify which tools or tool types must or should be included in each toolchain to mitigate identified risks, as well as how the toolchain components are to be integrated with each other.
- PO.3.2: Follow recommended security practices to deploy, operate, and maintain tools and toolchains.
- PO.3.3: Configure tools to generate artifacts of their support for security requirements and controls throughout the SDLC.

### PO.4 — Define and Use Criteria for Software Security Checks
Define and use criteria for performing security checks throughout the SDLC.

**Tasks:**
- PO.4.1: Define criteria for software security checks and track throughout the SDLC.
- PO.4.2: Implement processes, mechanisms, etc. to ensure that the criteria are met for each stage of the SDLC.

### PO.5 — Implement and Maintain Secure Environments for Software Development
Ensure all components of software development environments are strongly protected.

**Tasks:**
- PO.5.1: Separate and protect each environment involved in software development.
- PO.5.2: Secure and harden development endpoints (developer workstations, build systems, etc.) to perform development-related tasks using a risk-based approach.

## PS — Protect the Software

Protect all components of software from tampering and unauthorized access.

### PS.1 — Protect All Forms of Code from Unauthorized Access and Tampering

**Tasks:**
- PS.1.1: Store all forms of code — including source code, executable code, and configuration-as-code — based on the principle of least privilege so that only authorized personnel, tools, services, etc. have access.

### PS.2 — Provide a Mechanism for Verifying Software Release Integrity

**Tasks:**
- PS.2.1: Make software integrity verification information available to software acquirers.

### PS.3 — Archive and Protect Each Software Release

**Tasks:**
- PS.3.1: Securely archive the necessary files and supporting data (e.g., metadata, documentation, compose files) to be retained for each software release.
- PS.3.2: Collect, safeguard, maintain, and share provenance data for all components of each software release (e.g., SBOM, SCA results, code review records, tool outputs).

## PW — Produce Well-Secured Software

Minimize the number of security vulnerabilities in software releases.

### PW.1 — Design Software to Meet Security Requirements and Mitigate Security Risks

**Tasks:**
- PW.1.1: Use forms of risk modeling to identify and characterize risks to the software.
- PW.1.2: Track and maintain the risk model for each software project.
- PW.1.3: Where applicable, follow secure design principles such as: least privilege, defense in depth, fail securely, don't trust services, separation of duties, avoid security by obscurity, keep security simple, fix security issues correctly.

### PW.2 — Review the Software Design to Verify Compliance with Security Requirements

**Tasks:**
- PW.2.1: Have one or more independent experts review the software design, and have the development team remediate any identified issues.

### PW.4 — Reuse Existing, Well-Secured Software When Feasible

**Tasks:**
- PW.4.1: Acquire well-secured software components (e.g., software libraries, modules, middleware, frameworks) from commercial, open-source, and other third-party developers for use in the organization's software.
- PW.4.2: Create and maintain a curated repository of secured software components for internal use.
- PW.4.4: Verify that acquired commercial, open-source, and all other third-party software components comply with the requirements, as defined by the organization.

### PW.5 — Create Source Code by Adhering to Secure Coding Practices

**Tasks:**
- PW.5.1: Follow secure coding practices based on risk — e.g., input validation, output encoding, error handling, secure use of crypto, memory management.

### PW.6 — Configure the Compilation, Interpreter, and Build Processes to Improve Executable Security

**Tasks:**
- PW.6.1: Use compiler, interpreter, and build tools that offer features to improve executable security.
- PW.6.2: Determine which compiler, interpreter, and build tool features should be used and how each should be configured, then implement and use the approved configurations.

### PW.7 — Review and/or Analyze Human-Readable Code to Identify Vulnerabilities

**Tasks:**
- PW.7.1: Perform manual code reviews of security-critical components, using a documented, consistent process.
- PW.7.2: Perform automated scanning of code using SAST tools; address findings as appropriate.

### PW.8 — Test Executable Code to Identify Vulnerabilities

**Tasks:**
- PW.8.1: Use automated testing techniques to find vulnerabilities in executables — DAST, fuzzing, scanning.
- PW.8.2: Perform penetration testing to identify vulnerabilities.

### PW.9 — Configure Software to Have Secure Settings by Default

**Tasks:**
- PW.9.1: Define a baseline secure configuration for the software, and document deviations from that baseline.
- PW.9.2: Implement the baseline secure configuration in default settings, and require deviations from the baseline to be documented and approved.

## RV — Respond to Vulnerabilities

Identify residual vulnerabilities in software releases and respond appropriately.

### RV.1 — Identify and Confirm Vulnerabilities on an Ongoing Basis

**Tasks:**
- RV.1.1: Gather information from buyers/acquirers, users, and public sources on potential vulnerabilities in the software and third-party components that the software uses, and investigate all credible reports.
- RV.1.2: Review, analyze, and test the software's code to identify or better characterize vulnerabilities.
- RV.1.3: Have a policy that addresses vulnerability disclosure.

### RV.2 — Assess, Prioritize, and Remediate Vulnerabilities

**Tasks:**
- RV.2.1: Analyze each vulnerability to gather sufficient information about risk to support prioritization decisions.
- RV.2.2: Plan and implement risk-based remediation of vulnerabilities.

### RV.3 — Analyze Vulnerabilities to Identify Their Root Causes

**Tasks:**
- RV.3.1: Analyze vulnerabilities to identify their root causes.
- RV.3.2: Analyze the root causes over time to identify patterns and determine if process or technology improvements are needed.
- RV.3.3: Review the software for similar vulnerabilities to remediate them proactively.
